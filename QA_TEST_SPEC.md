# ТЗ на тестирование: Parser Tender XLSX

## 1. Контекст и цель

**Проект:** Парсер тендерных таблиц XLSX — автоматическое извлечение структурированных данных из Excel-файлов тендеров и отправка в API.

**Цель тестирования:** Гарантировать корректность извлечения данных при любом разумном формате входного файла, включая «грязные» и нестандартные случаи.

---

## 2. Архитектура и границы системы

### Основные модули

| Модуль | Путь | Назначение | Сложность тестирования |

| **Gemini Module** | `app/gemini_module/` | LLM-постобработка позиций | ★★★★ (сетевой вызов, недетерминированность) |

## 4. Уровни тестирования

### 4.1 Unit-тесты (`tests/unit/`, `app/tests/`)

**Что:** Изолированные функции без I/O, без сети, без файловой системы.

**Как:** `openpyxl.Workbook()` для создания синтетических листов в памяти; `monkeypatch` для подмены зависимостей.

**Маркер:** `@pytest.mark.unit`

**Целевое покрытие:** ≥ 80% строк в `app/excel_parser/`

**Ключевые модули для unit-покрытия:**

- `app/excel_parser/read_headers.py`
- `app/excel_parser/read_lots_and_boundaries.py`
- `app/excel_parser/read_contractors.py`
- `app/excel_parser/parse_contractor_row.py`
- `app/excel_parser/get_items_dict.py`
- `app/excel_parser/get_lot_positions.py`
- `app/excel_parser/get_additional_info.py`
- `app/excel_parser/read_executer_block.py`
- `app/excel_parser/sanitize_text.py`
- `app/excel_parser/postprocess.py`
- `app/excel_parser/get_proposals.py`

### 4.2 Интеграционные тесты (`tests/integration/`)

**Что:** Полный pipeline парсинга реального/синтетического .xlsx файла; взаимодействие модулей.

**Как:** Читать `.xlsx` из `tests/fixtures/excel/`; сравнивать с golden JSON из `tests/fixtures/json/`.

**Маркер:** `@pytest.mark.integration`

**Запуск:** `make test-integration` (в CI или отдельно; требуют наличия fixture-файлов)

### 4.3 Контрактные тесты (`tests/contract/`)

**Что:** Структура выходного JSON соответствует ожидаемой схеме (обязательные ключи, типы данных).

**Как:** JSON Schema validation (`jsonschema`) или Pydantic-модели против реального вывода.

**Маркер:** `@pytest.mark.unit` (если схема статичная) или `@pytest.mark.integration`

### 4.4 Gemini-тесты

**Что:** Корректная обработка ответов Gemini: парсинг JSON из текстового ответа, обработка ошибок, retry-логика.

**Как:** Мок-ответы из `tests/fixtures/gemini/`; тесты с реальным API только при явном указании.

**Маркеры:** `@pytest.mark.unit` для мок-тестов; `@pytest.mark.gemini` для реальных вызовов.

## 5. Детальные требования по функциональным блокам

### 5.1 Парсинг шапки тендера (`read_headers`)

**Что проверять:**
- Извлечение «Предмет тендера» → `tender_id` + `tender_title` (разбивка по первому пробелу после `№`)
- Извлечение «Объект» → `tender_object`
- Извлечение «Адрес объекта» → `tender_address`
- **Нежёсткое сравнение ключей:** игнорировать ведущие/завершающие пробелы, двоеточия, разный регистр
- Диапазон сканирования: только строки 3–5 (индексы 1-based)
- Возвращать `None` для отсутствующих полей (не бросать исключение)

**Граничные случаи:**
- Шапка в строках вне диапазона 3–5
- Числовые значения вместо строковых
- Только символ `№` без номера
- Дублирующиеся ключи (использовать последний)

### 5.2 Лоты (`read_lots_and_boundaries`)

**Что проверять:**
- Строка «Лот №…» начинает новый контекст лота
- Каждый новый «Лот №…» закрывает предыдущий и открывает новый
- Корректное определение границ (начало/конец) каждого лота
- Один лот, несколько лотов (3+), отсутствие лотов

**Граничные случаи:**
- «Лот №» в разных колонках
- «Лот №» с пробелами вокруг числа
- Файл без строк «Лот №»

### 5.3 Подрядчики (`read_contractors`, `parse_contractor_row`)

**Что проверять:**
- Поиск подрядчиков в строке 6 (0-based row index 5) по всем колонкам
- Исключать из кандидатов строки «Наименование контрагента» и «Расчетная стоимость»
- Для каждого подрядчика: имя, диапазон колонок (start_col..end_col)
- Метаданные подрядчика из первых 4 строк блока КП: Наименование, ИНН, Адрес, Статус аккредитации

**Граничные случаи:**
- Один подрядчик
- Три и более подрядчиков
- Разные наборы колонок у подрядчиков
- Лишние пустые колонки между подрядчиками

### 5.4 Позиции (items) (`get_lot_positions`, `get_items_dict`)

**Что проверять:**
- Парсинг прекращается на строке, содержащей «Дополнительная информация» (case-insensitive)
- Доп. поля в items:
  - «Предлагаемое количество» → `suggested_quantity`
  - «Общее кол-во» → `organizer_quantity_total_cost` (или соответствующий ключ)
  - «Стоимость всего за объемы заказчика» → соответствующий ключ
  - Если значение пусто — поле **не включать** в результат
- Строки «ИТОГО …» и «В том числе НДС …» добавлять как items
- Анализ до первой полностью пустой строки (за которой идёт блок доп. информации)

**Граничные случаи:**
- Позиции без числовых значений
- Позиции с нулевыми значениями (0 vs отсутствие)
- Несколько «ИТОГО» строк
- «Дополнительная информация» в середине позиций (аномалия)

### 5.5 Блок «Дополнительная информация» (`get_additional_info`)

**Что проверять:**
- Ключи в первом столбце (0)
- Значения берутся из колонок каждого подрядчика
- `NaN` → сохранять как `null` в JSON
- Парсинг прекращается при появлении строк исполнителя: «Исполнитель:», «тел:», «Дата составления …»

**Граничные случаи:**
- Блок полностью отсутствует
- Все значения NaN
- Ключи с пробелами/двоеточиями

### 5.6 Блок исполнителя (`read_executer_block`)

**Что проверять:**
- Извлечение имени исполнителя (по префиксу «Исполнитель:»)
- Извлечение телефона (по префиксу «тел:»)
- Извлечение даты (по префиксу «Дата составления»)
- Блок — последние строки таблицы перед пустой строкой (или конец листа)

**Граничные случаи:**
- Неполный блок (только имя, без телефона)
- Блок отсутствует
- Дополнительные строки после блока

### 5.7 Постобработка (`postprocess`)

**Что проверять:**
- Очистка пустых/нулевых значений
- Нормализация типов данных
- Сохранение структуры при пустом вводе

## 6. Стратегия мокирования

### 6.1 Gemini API

**Подход A — monkeypatch (рекомендуется для unit):**
```python
monkeypatch.setattr("app.gemini_module.processor.GeminiProcessor.generate", lambda *a: mock_response)
```

**Подход B — VCR-like (для интеграционных):**
Хранить JSON-файлы реальных ответов в `tests/fixtures/gemini/`. При тесте читать файл и возвращать как фиктивный ответ.

**Подход C — интерфейс-адаптер:**
Если `GeminiProcessor` реализует протокол/интерфейс, в тестах подставлять `FakeGeminiProcessor` с детерминированным ответом.

### 6.2 HTTP (Go API, JSON to Server)

```python
# requests
monkeypatch.setattr("requests.post", lambda *a, **kw: FakeResponse(200, {"status": "ok"}))

# httpx async
monkeypatch.setattr("httpx.AsyncClient.post", AsyncMock(return_value=FakeResponse(...)))
```

### 6.3 Celery

```python
@pytest.fixture(autouse=True)
def celery_eager_mode(monkeypatch):
    """Выполнять задачи синхронно, без брокера."""
    monkeypatch.setenv("CELERY_TASK_ALWAYS_EAGER", "true")
```

### 6.4 База данных (PostgreSQL / pgvector)

Для unit-тестов: мок SQLAlchemy session через `unittest.mock.MagicMock`.
Для интеграционных: тестовая БД в Docker или SQLite с in-memory режимом.

### 6.5 Даты и случайность

```python
from unittest.mock import patch
from datetime import datetime

@pytest.fixture
def fixed_datetime():
    with patch("app.excel_parser.read_executer_block.datetime") as m:
        m.now.return_value = datetime(2024, 1, 15, 12, 0, 0)
        yield m
```

## 8. Критерии приёмки (Definition of Done)

### Покрытие кода

| Модуль | Минимальное покрытие |
|--------|---------------------|
| `app/excel_parser/` | ≥ 80% |
| `app/gemini_module/` | ≥ 60% |
| `app/go_module/` | ≥ 50% |
| Общее по `app/` | ≥ 65% |

### Обязательные условия

- [ ] Все юнит-тесты проходят за < 30 секунд (`make test-fast`)
- [ ] `make test` проходит без флага `--ignore` для интеграционных тестов (они пропускаются через маркеры, а не через `--ignore`)
- [ ] Ни один тест не делает реальных сетевых вызовов без маркера `@pytest.mark.integration` или `@pytest.mark.gemini`
- [ ] Все тесты детерминированы: повторный запуск даёт тот же результат
- [ ] `make check` проходит без ошибок (форматирование кода)
- [ ] CI (`make test-coverage`) проходит в GitHub Actions

### Новый тест считается принятым, если

1. Имеет явный маркер (`@pytest.mark.unit`, `@pytest.mark.integration` и т.д.)
2. Описывает конкретный бизнес-кейс (не абстрактный «тест функции X»)
3. Следует паттерну Arrange / Act / Assert
4. Не зависит от порядка запуска других тестов
5. Проходит локально командой `make test-fast` (или `make test-integration` для интеграционных)

## 9. Рекомендации по тестируемости кода

> Это рекомендации, а не обязательные изменения. Применять только если изменение безопасно и не ломает существующую логику.

1. **Dependency Injection для Gemini:** вместо создания `GeminiProcessor` внутри функции — принимать его как параметр. Это упрощает мокирование без `monkeypatch`.

2. **Pure functions для парсинга:** функции типа `read_headers(ws)` уже близки к pure. Убедитесь, что они не имеют скрытых side-effects (запись в файл, логирование как условие, глобальное состояние).

3. **Фабричный метод для Excel Workbook:** вместо `pd.read_excel(path)` напрямую — принимать уже загруженный `DataFrame` или `openpyxl.Worksheet`. Это позволяет тестировать без файловой системы.

4. **Константы для строковых маркеров:** «Дополнительная информация», «Лот №», «Исполнитель:» и т.д. — должны быть определены в `app/constants.py`. Использовать константы в тестах, а не дублировать строки.
